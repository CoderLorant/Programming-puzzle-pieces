12. Emergence

Getting Clean via Emergent Design
"Many of us feel that Kent Beck's four rules of Simple Design are of significant help in creating well-designed software. According to Kent, a design is "simple" if it follows these rules:

Runs all the tests
Contains no duplication
Expresses the intent of the programmer
Minimizes the number of classes and methods

The rules are given in order of importance."

Simple Design Rule 1: Runs All the Tests
"A system might have a perfect design on paper, but if there is no simple way to verify that the system actually works as intended, then all the paper effort is questionable."

"Fortunately, making our systems testable pushes us toward a design where our classes are small and single purpose. It's just easier to test classes that conform to the SRP."

"Tight coupling makes it difficult to write tests. So, similarly, the more tests we write, the more we use principles like DIP and tools like dependency injection, interfaces, and abstraction to minimize coupling."

"Writing tests leads to better design."

Simple Design Rules 2-4: Refactoring
"Once we have tests, we are empowered to keep our code and classes clean. [...] The fact that we have these tests eliminates the fear that cleaning up the code will break it!"

"We can increase cohesion, decrease coupling, separate concerns, modularize system concerns, shrink our functions and classes, choose better names, and so on. This is also where we apply the final three rules of simple design: Eliminate duplication, ensure expressiveness, and minimize the number of classes and methods."

No Duplication
"Duplication is the primary enemy of a well-designed system. It represents additional work, additional risk, and additional unnecessary complexity. [...] Lines of code that look exactly alike are, of course, duplication."

"Creating a clean system requires the will to eliminate duplication, even in just a few lines of code."

"The TEMPLATE METHOD pattern is a common technique for removing higher-level duplication."

Expressive
"The majority of the cost of a software project is in long-term maintenance."

"Therefore, code should clearly express the intent of its author. The clearer the author can make the code, the less time others [including you after few weeks or months] will have to spend understanding it."

"You can express yourself by choosing good names."

"You can also express yourself by keeping your functions and classes small."

"You can also express yourself by using standard nomenclature. Design patterns, for example, are largely about communication and expressiveness."

"Well-written unit tests are also expressive. A primary goal of tests is to act as documentation by example."

"But the most important way to be expressive is to try."

"take care of what you've created. Care is a precious resource."

Minimal Classes and Methods
"High class and method counts are sometimes the result of pointless dogmatism."

"Remember, however, that this rule is the lowest priority of the four rules of Simple Design. So, although it's important to keep class and function count low, it's more important to have tests, eliminate duplication, and express yourself."
